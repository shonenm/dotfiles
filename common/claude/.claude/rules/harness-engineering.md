# Harness Engineering

Claude Code ハーネス（設定・フック・スキル・エージェント・ルール）を体系的に設計・最適化する規律。
ハーネスの質がセッション全体の生産性を決定する。個々のタスク実装と同等の注意を払う。

## コンポーネント体系

| コンポーネント | グローバル (`~/.claude/`) | プロジェクト (`.claude/`) | 役割 |
|---------------|-------------------------|------------------------|------|
| rules/ | 個人の作業方針・価値観 | チーム・プロジェクト規約 | Claude の振る舞いを制約する指示 |
| skills/ | `_` prefix、汎用ワークフロー | prefix なし、プロジェクト固有 | ユーザーが起動する定型作業 |
| agents/ | 汎用サブエージェント | プロジェクト特化エージェント | Agent ツール経由で呼ばれる専門家 |
| hooks/ | フィードバックループ・通知 | プロジェクト固有チェック | イベント駆動の自動処理 |
| settings.json | グローバル権限 | プロジェクトメタデータ・権限 | ハーネスの動作パラメータ |
| settings.local.json | -- | 個人のローカル設定 | 個人権限・MCP・hooks |
| CLAUDE.md | -- | プロジェクト指示書 | コンテキスト注入の起点 |

## 設計原則

- **レイヤリング**: グローバル(dotfiles) → プロジェクト → ローカル。汎用は上位、固有は下位に配置
- **フィードバックループ**: PostToolUse フックで型チェック・lint・テストの即時フィードバックを返す
- **最小権限**: deny でデフォルト拒否、allow で明示許可。secrets は `Read(.env.*)` で deny
- **コンテキスト効率**: ルールは簡潔に保ち、詳細な手順はスキル/エージェントに委譲する
- **進化的改善**: `/_harness-audit` で定期的にハーネスを監査し、改善サイクルを回す
- **関心の分離**: 1 ルール = 1 関心事、1 スキル = 1 ワークフロー、1 エージェント = 1 専門領域

## フック設計パターン

| イベント | 用途 | 例 |
|---------|------|-----|
| Stop | 自律ループ制御・完了判定 | ralph-stop-hook (タスク進捗ベースの継続/停止) |
| PostToolUse | 即時品質フィードバック | ralph-backpressure (Write/Edit 後に tsc/lint/test) |
| PreCompact | コンテキスト圧縮時の状態保存 | ralph-precompact (ループ状態ファイルのパス注入) |
| SessionStart | セッション初期化 | ralph-session-context (プロジェクト構造の自動収集) |
| CwdChanged | 環境適応 | claude-cwd-changed (.envrc の自動読み込み) |
| Notification | 通知連携 | 外部通知システムへの転送 |

## スキル命名規則

- グローバル (dotfiles): `_` prefix (`_commit`, `_pr`, `_harness-audit`)
- プロジェクト固有: prefix なし (`create-feature`, `fix-issue`)
- 汎用化して import する際は `_` prefix を付与

## 改善の判断基準

ハーネスを変更する前に以下を確認する:
- この変更は複数セッションにわたって価値を持つか (一時的なら変更しない)
- グローバルかプロジェクト固有か (配置レイヤーの選択)
- 既存コンポーネントの拡張で対応できるか (新規作成より既存改善を優先)
